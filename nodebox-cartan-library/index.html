The Cartan Node Library, on the web

John Cartan’s Node Library is the largest community contribution to NodeBox 3 — 191 nodes for geometry, color, text, math, lists, and data, built over a decade of daily practice. This experiment ports version 3.6 to the unified web NodeBox engine: the original .ndbx document is converted to JSON, its Python and Clojure helper functions are re-implemented in JavaScript, and every node’s original demo is evaluated by the engine — offline for the thumbnails below, and live in your browser when you open a node.

How the port works ↓ · source & conversion pipeline

About this port

Cartan’s library ships as one 188 MB NodeBox 3 document. Each of its nodes is a subnetwork — a graph built from the ~130 standard NodeBox nodes plus a handful of Python and Clojure helper functions. Because NodeBox files copy a subnetwork wholesale every time it is reused, the document holds around 630,000 node instances; only about 109,000 of them are canonical definitions, the rest are the per-node demos you see as thumbnails above.

The pipeline (see tools/ and doc/DECISIONS.md):

  1. Rebuild the NodeBox 3 standard library in JavaScript. Port metadata (names, defaults, menus) is generated from the official .ndbx library files; the ~130 function bodies were ported by hand from the Java and Jython sources (engine/n3lib.js). Path boolean operations use a vendored polybooljs; text is drawn as native text with approximated Helvetica metrics instead of AWT font outlines.
  2. Convert the document. A parser resolves NodeBox’s port-forwarding connections, sparse default names, and write-through published ports into the unified engine’s JSON format — one file per node, demo included when it fits a size budget.
  3. Evaluate everything, headless. The same engine that runs in this page runs under Node to execute all 191 demos and pre-render the SVG thumbnails. 186 nodes evaluate end-to-end; the five that don’t need a local filesystem, installed fonts, or NodeBox application introspection — things a browser deliberately can’t do.

Could the Python nodes be ported to JavaScript automatically?

Mostly, yes — but not by a transpiler. The 19 helper scripts (17 Python, 2 Clojure) are small (12 lines to ~200 lines) and lean on NodeBox’s Java classes (Path, Point, AWT fonts, ImageIO), so mechanical Python→JS translation (Transcrypt, Brython) would fail exactly where it matters and drag a runtime along. What is automatic in 2026 is LLM translation: each script was ported by AI against the original sources, with the engine’s headless evaluation of all 191 demos as the referee. Thirteen ported cleanly (poisson sampling, Perlin noise, treemaps, Hilbert curves, convex hulls, date formatting…); the other six are impossible in a sandboxed browser by design (directory listing, writing temp files, reading installed fonts, sampling local images, app introspection) and are flagged unsupported above.

Fidelity notes

  • Text nodes (textpath and everything built on it) draw native text with approximate metrics, not font outlines — layouts are faithful, letterforms are your system’s. Point-level access to glyph outlines degrades to the text’s bounding box.
  • Random numbers are deterministic per seed but use a different generator than Java’s — stochastic nodes produce the same kind of picture, not the same picture.
  • Path booleans (compound) flatten curves before combining, like Java’s Area on flattened paths.
